Key Findings
1. The historical Paper 2 review process was not just a prompt for general peer review. It was a structured pipeline with distinct artifacts:
   - formalism checkpoint
   - section review
   - dependency / blast-radius analysis
   - triage verdict
   - targeted spike commissions
   - revision and escalation into later-paper backlog

2. The formalism layer is load-bearing.
   The most important missing piece in the desktop-only reconstruction was the global checkpoint stage represented by FORMALISM_CHECKPOINT.md.

3. Global inconsistency and local progress can coexist.
   The warp-factor inconsistency flagged in the checkpoint was not globally resolved paper-wide. However, section-level work proceeded under an explicit local convention lock for SC2, and that local lock was treated as sufficient for §7 review and draft promotion.

4. Paper 3 emerged partly as an escalation path.
   The review process did not only produce corrections. It also identified unresolved structural issues that were better migrated into a successor-paper backlog.

Recovered Operational Rules
- Run a global formalism checkpoint before promoting section-level conclusions to manuscript-level confidence.
- Allow section review to proceed only when the local convention is explicit and locked.
- Carry unresolved convention drift into triage instead of silently ignoring it.
- Use targeted spike investigations when a blocking question is too narrow or technical for the main review flow.
- Distinguish:
  - fix now in current paper
  - note but do not block
  - escalate to successor-paper backlog

Recovered Review Pipeline
1. Intake and freeze the target section or manuscript state.
2. Run a global formalism checkpoint.
3. Run section-level technical reviews.
4. Generate dependency / blast-radius analysis (T-files).
5. Issue a triage verdict.
6. Commission spikes for unresolved blocking questions when needed.
7. Produce revision program and author-response commitments.
8. Escalate unresolved structural findings into later-paper backlog.

External Tool Scan
An external search was run for research program management, structured review, and stage-gate portfolio tools.

Most relevant categories and examples:

- Research administration / workflow systems
  Cayuse, Kuali Research, Fibi, Insight, Researchweb

- Assessment / panel review systems
  Tactiv / Enquire, Dapple

- R&D portfolio / stage-gate systems
  Planisware, ITONICS, GenSight, Cora

Useful patterns observed in those tools:
- configurable routing and approvals
- reviewer panels and structured scoring
- audit trails and status transparency
- configurable stage gates
- dashboarded portfolio visibility
- follow-up and reporting workflows

Critical gaps relative to the CR use case:
- no native formalism-checkpoint layer
- no VERIFIED / UNTESTED / MISSING / CONDITIONAL physics-status ontology
- no built-in dependency T-file / blast-radius analysis
- no spike-commission mechanism for adjudicating narrow blocking questions
- no successor-paper backlog logic as a first-class outcome of review

Decision
Build a custom version.

Rationale
- The internal CR review process already contains a domain-specific governance model that off-the-shelf tools do not represent.
- Existing products cover generic workflow mechanics well, but not the mathematics- and formalism-sensitive logic that made the Paper 2 review productive.
- The most reliable path is to preserve the proven CR artifact structure and borrow only generic workflow patterns from external systems.
