# Preliminary Report Tools

**Week 2 Deliverable:** SR → Preliminary Report → ACR AC Enrichment → Consistency Analysis

This directory contains tools for generating, enriching, and analyzing preliminary radiology reports extracted from DICOM Structured Reports (SRs).

---

## Overview

The preliminary report pipeline consists of three stages:

1. **Extraction** (dicom-ecosystem): Parse TID 1500 SR files into structured JSON
2. **Enrichment** (this directory): Link procedure codes to ACR Appropriateness Criteria topics
3. **Consistency Analysis** (../packages/report-consistency): Validate measurements and flag potential issues

---

## Quick Start

### End-to-End Workflow

```bash
# Run the complete Week 2 prototype pipeline
./tools/preliminary-report/example_workflow.sh
```

This script:
- Extracts preliminary report from `test_tid1500_sr.dcm`
- Enriches with ACR AC topic metadata
- Displays summary output

### Manual Step-by-Step

#### Step 1: Extract Preliminary Report

```bash
cd ../dicom-ecosystem
python sr-tools/preliminary_report_from_sr.py \
  --sr-file sr-tools/test-data/test_tid1500_sr.dcm \
  --json-output /tmp/preliminary_report.json
```

**Output:** Structured JSON with:
- Report metadata (IDs, study UID, modality, procedure code)
- Measurement groups (tracking IDs, finding sites, measurements with units)

#### Step 2: Enrich with ACR AC Metadata

```bash
cd ai-imaging-platform
python tools/preliminary-report/enrich_with_acr_ac.py \
  --report /tmp/preliminary_report.json \
  --output /tmp/enriched_report.json
```

**Output:** Enhanced JSON with:
- Original preliminary report content
- `acr_topics` array with matched ACR AC topics, relevance scores, and PDF paths
- `_enrichment_metadata` tracking mapping status

#### Step 3: Analyze Consistency (Phase 1)

```bash
python packages/report-consistency/consistency_engine.py \
  --report /tmp/enriched_report.json \
  --output /tmp/consistency_profile.json
```

**Output:** Consistency profile with:
- Dimension scores (internal_measurements, longitudinal, follow_up, lexical)
- Structured flags with severity, evidence, and suggested actions
- Metadata about analysis phase and scope

## Tools Reference

### `enrich_with_acr_ac.py`

**Purpose:** Map procedure codes (LOINC/CPT) to ACR Appropriateness Criteria topics.

**Usage:**
```bash

  --report <preliminary_report.json> \
  --output <enriched_report.json> \
```

**Options:**
- `--report`: Path to preliminary report JSON (or stdin if omitted)

## Consistency Analysis (Phase 1)

### Current Scope

**Internal Measurement Consistency:**
- Missing units detection
- Negative value checks
- Out-of-range value warnings (liver span, wall thickness, etc.)
- Units mismatch detection (cm vs mm)

### Scoring Algorithm

```
internal_measurements_score = 1.0
for each flag:
  if severity == "critical": score -= 0.3
  if severity == "warning":  score -= 0.1
  if severity == "info":     score -= 0.02

score = max(0.0, min(1.0, score))
```

### Future Phases

- **Phase 2 (Week 3+):** Longitudinal consistency
  - Compare measurements across time
  - Growth/shrinkage detection
  - Prior recommendation tracking
  
- **Phase 3 (Week 4+):** Follow-up recommendations
  - ACR AC guideline alignment
  - Missing follow-up closure detection
  
- **Phase 4 (Quarter 1):** Lexical consistency
  - RadLex term usage
  - Template adherence
  - Hedging language detection

## Troubleshooting

### "No ACR AC topics found"

**Cause:** Procedure code not in `LOINC_TO_ACR_MAPPING`.

**Solution:**
1. Check procedure code in preliminary report JSON: `jq '.procedure' <report.json>`
2. Add mapping to `enrich_with_acr_ac.py` (lines 43-71)
3. Or wait for GraphRAG integration (Week 3) for comprehensive coverage

### "ACR index directory not found"

**Cause:** ACR AC index missing or path incorrect.

**Solution:**
```bash
# Verify index exists
ls databases/acr-criteria/acr_ac_index.json

# Or rebuild index
python tools/acr-downloader/acr_ac_index_builder.py
```

### Consistency engine reports false positives

**Cause:** Measurement sanity limits may be too strict for edge cases.

**Solution:**
- Review limits in `consistency_engine.py` (lines 73-89)
- Adjust thresholds based on clinical input
- Or add measurement-specific exceptions

def map_procedure_to_acr_topics(
    procedure_code: Optional[Dict[str, str]],
    acr_grouped: Dict[Any, Any],
) -> List[Dict[str, Any]]:
    """Map a procedure code (LOINC/CPT/RPID) to ACR AC topics.
    
    Returns a list of ACR topic objects enriched with a relevance_score.
    """
    if not procedure_code:
        return []
    
    code_value = procedure_code.get("code", "").strip()
    code_system = procedure_code.get("system", "").strip().upper()
    
    # For Week 2 prototype: only handle LOINC codes via LOINC_TO_ACR_MAPPING
    if code_system not in ("LN", "LOINC"):
        # Future: add CPT, RPID, RadLex mappings
        return []
    
    if code_value not in LOINC_TO_ACR_MAPPING:
        # No mapping available yet
        return []
    
    results: List[Dict[str, Any]] = []
    
    for panel, topic_slug, relevance in LOINC_TO_ACR_MAPPING[code_value]:
        # Look up topic in ACR index
        key = (panel, topic_slug)
        if key not in acr_grouped:
            continue
        
        topic_obj = acr_grouped[key].copy()
        topic_obj["relevance_score"] = relevance
        results.append(topic_obj)
    
    # Sort by relevance descending
    results.sort(key=lambda x: x.get("relevance_score", 0), reverse=True)
    
    return results

"""Minimal Consistency Engine (Week 2 Prototype)

Implements Phase 1 consistency checks focused on internal measurement validation.

This module analyzes preliminary report JSON (from the SR extraction pipeline) and
emits a Report Consistency Profile with dimension scores and structured flags.

Usage (from repository root):

    python packages/report-consistency/consistency_engine.py \
      --report /tmp/enriched_report.json \
      --output /tmp/consistency_profile.json

Week 2 Scope (Phase 1):
- Internal measurement consistency
- Basic sanity checks (units, ranges, completeness)
- No longitudinal analysis (requires prior reports)

Future Phases:
- Longitudinal consistency (requires prior report tracking)
- Follow-up recommendation analysis
- Lexical consistency checks
"""

from __future__ import annotations

import argparse
import json
import sys
from dataclasses import dataclass, asdict, field
from pathlib import Path
from typing import Any, Dict, List, Optional


@dataclass
class ConsistencyFlag:
    """A structured flag indicating a potential consistency issue."""
    
    id: str  # Stable rule identifier (e.g., "meas_001_missing_units")
    severity: str  # "info", "warning", "critical"
    dimension: str  # Which consistency dimension triggered this
    message: str  # Human-readable explanation
    evidence: Dict[str, Any]  # Structured references to findings/measurements
    suggested_action: Optional[str] = None


@dataclass
class ConsistencyScores:
    """Consistency scores by dimension (0.0 = poor, 1.0 = excellent)."""
    
    internal_measurements: float = 1.0
    longitudinal: float = 1.0  # Not scored in Phase 1
    follow_up: float = 1.0  # Not scored in Phase 1
    lexical: float = 1.0  # Not scored in Phase 1


@dataclass
class ConsistencyProfile:
    """Complete consistency analysis result for a single report."""
    
    report_id: Optional[str]
    study_uid: Optional[str]
    modality: Optional[str]
    body_region: Optional[str]
    consistency_scores: ConsistencyScores
    flags: List[ConsistencyFlag] = field(default_factory=list)
    metadata: Dict[str, Any] = field(default_factory=dict)


# Phase 1 measurement sanity thresholds (conservative defaults)
# These can be refined based on clinical input and statistical analysis.
MEASUREMENT_SANITY_LIMITS = {
    # Body region measurements (typical max in cm)
    "liver_span": {"min": 5.0, "max": 30.0, "units": ["cm"]},
    "spleen_length": {"min": 5.0, "max": 25.0, "units": ["cm"]},
    "kidney_length": {"min": 7.0, "max": 20.0, "units": ["cm"]},

    # Nodule/lesion diameter (typical max in cm)
    "nodule_diameter": {"min": 0.1, "max": 20.0, "units": ["cm", "mm"]},
    "lesion_diameter": {"min": 0.1, "max": 30.0, "units": ["cm", "mm"]},
    
    # Generic size measurement (very broad)
    "generic_size": {"min": 0.01, "max": 100.0, "units": ["cm", "mm"]},
}


def _extract_measurement_key(name_code: Dict[str, str]) -> Optional[str]:
    """Extract a normalized measurement key for sanity checking.
    
    Maps RadLex or LOINC codes to generic measurement types.
    For Week 2, use simple keyword matching on meaning.
    """
    meaning = name_code.get("meaning", "").lower()
    
    if "liver" in meaning and "span" in meaning:
        return "liver_span"
    if "gallbladder" in meaning and "wall" in meaning and "thickness" in meaning:
        return "gallbladder_wall_thickness"
    if "wall" in meaning and "thickness" in meaning:
        return "wall_thickness"
    if "nodule" in meaning and ("diameter" in meaning or "size" in meaning):
        return "nodule_diameter"
    if "lesion" in meaning and ("diameter" in meaning or "size" in meaning):
        return "lesion_diameter"
    
    # Default: generic size
    if any(kw in meaning for kw in ["size", "diameter", "length", "width", "span"]):
        return "generic_size"
    
    return None


def check_measurement_sanity(
    measurement: Dict[str, Any],
    group_context: Dict[str, Any],
) -> List[ConsistencyFlag]:
    """Check a single measurement for basic sanity (units, range).
    
    Args:
        measurement: Measurement dict with name, value, units
        group_context: Parent measurement group for context (tracking_id, finding_site)
    
    Returns:
        List of ConsistencyFlag objects (empty if all checks pass)
    """
    flags: List[ConsistencyFlag] = []
    
    name = measurement.get("name", {})
    value = measurement.get("value")
    units = measurement.get("units", {})
    
    name_meaning = name.get("meaning", "Unknown measurement")
    unit_code = units.get("code", "")
    unit_meaning = units.get("meaning", "")
    
    tracking_id = group_context.get("tracking_id", "unnamed_group")
    finding_site = group_context.get("finding_site", {}).get("meaning", "unspecified site")
    
    # Check 1: Missing units
    if not unit_code:
        flags.append(
            ConsistencyFlag(
                id="meas_001_missing_units",
                severity="warning",
                dimension="internal_measurements",
                message=f"Measurement '{name_meaning}' missing units",
                evidence={
                    "measurement_name": name_meaning,
                    "value": value,
                    "group": tracking_id,
                    "finding_site": finding_site,
                },
                suggested_action="Verify units and add MeasurementUnitsCodeSequence",
            )
        )
    
    # Check 2: Value range sanity
    if value is not None:
        try:
            val = float(value)
            
            # Negative measurements (generally invalid)
            if val < 0:
                flags.append(
                    ConsistencyFlag(
                        id="meas_002_negative_value",
                        severity="critical",
                        dimension="internal_measurements",
                        message=f"Measurement '{name_meaning}' has negative value: {val}",
                        evidence={
                            "measurement_name": name_meaning,
                            "value": val,
                            "units": unit_code,
                            "group": tracking_id,
                        },
                        suggested_action="Verify measurement accuracy; negative values are typically invalid",
                    )
                )
            
            # Check against sanity limits
            meas_key = _extract_measurement_key(name)
            if meas_key and meas_key in MEASUREMENT_SANITY_LIMITS:
                limits = MEASUREMENT_SANITY_LIMITS[meas_key]
                min_val = limits["min"]
                max_val = limits["max"]
                expected_units = limits["units"]
                
                # Warn if out of expected range
                if val < min_val or val > max_val:
                    flags.append(
                        ConsistencyFlag(
                            id=f"meas_003_out_of_range_{meas_key}",
                            severity="warning",
                            dimension="internal_measurements",
                            message=(
                                f"Measurement '{name_meaning}' value {val} {unit_code} "
                                f"outside typical range ({min_val}–{max_val} {'/'.join(expected_units)})"
                            ),
                            evidence={
                                "measurement_name": name_meaning,
                                "value": val,
                                "units": unit_code,
                                "expected_range": f"{min_val}-{max_val}",
                                "expected_units": expected_units,
                                "group": tracking_id,
                            },
                            suggested_action=(
                                "Verify measurement accuracy and units; "
                                "if correct, consider documenting rationale for unusual value"
                            ),
                        )
                    )
                
                # Warn if units mismatch
                if unit_code and unit_code not in expected_units:
                    flags.append(
                        ConsistencyFlag(
                            id=f"meas_004_units_mismatch_{meas_key}",
                            severity="info",
                            dimension="internal_measurements",
                            message=(
                                f"Measurement '{name_meaning}' uses units '{unit_code}'; "
                                f"expected: {'/'.join(expected_units)}"
                            ),
                            evidence={
                                "measurement_name": name_meaning,
                                "value": val,
                                "units": unit_code,
                                "expected_units": expected_units,
                                "group": tracking_id,
                            },
                            suggested_action="Convert to expected units or document rationale",
                        )
                    )
        
        except (TypeError, ValueError):
            flags.append(
                ConsistencyFlag(
                    id="meas_005_non_numeric_value",
                    severity="critical",
                    dimension="internal_measurements",
                    message=f"Measurement '{name_meaning}' has non-numeric value: {value}",
                    evidence={
                        "measurement_name": name_meaning,
                        "value": str(value),
                        "group": tracking_id,
                    },
                    suggested_action="Ensure NumericValue is a valid float",
                )
            )
    
    return flags

    """Analyze a preliminary report for consistency issues.
    
    Phase 1 (Week 2):
    - Internal measurement consistency only
    - No longitudinal analysis (requires prior reports)
    - No follow-up or lexical analysis
    
    Args:
        report: Preliminary report JSON (from preliminary_report_from_sr.py)

    study_uid = report.get("study_uid")
    modality = report.get("modality")
    body_region = report.get("body_region")
    
    measurement_groups = report.get("measurement_groups", [])

    for group in measurement_groups:
        group_flags = check_measurement_group(group)
        all_flags.extend(group_flags)

    # Simple scoring: start at 1.0, deduct points for flags
    # - Critical flag: -0.3
    # - Warning flag: -0.1
    # - Info flag: -0.02
    internal_score = 1.0
    for flag in all_flags:
        if flag.dimension == "internal_measurements":
            if flag.severity == "critical":
                internal_score -= 0.3
            elif flag.severity == "warning":
                internal_score -= 0.1
            elif flag.severity == "info":
                internal_score -= 0.02
    
    internal_score = max(0.0, min(1.0, internal_score))
    
    scores = ConsistencyScores(
        internal_measurements=internal_score,
        longitudinal=1.0,  # Not analyzed in Phase 1
        follow_up=1.0,  # Not analyzed in Phase 1
        lexical=1.0,  # Not analyzed in Phase 1
    )
    
    metadata = {
        "phase": "1",
        "scope": "internal_measurements_only",
        "total_flags": len(all_flags),
        "critical_flags": sum(1 for f in all_flags if f.severity == "critical"),
        "warning_flags": sum(1 for f in all_flags if f.severity == "warning"),
        "info_flags": sum(1 for f in all_flags if f.severity == "info"),
        "measurement_groups_analyzed": len(measurement_groups),
    }
    
    profile = ConsistencyProfile(
        report_id=report_id,
        study_uid=study_uid,
        modality=modality,
        body_region=body_region,
        consistency_scores=scores,
        flags=all_flags,
        metadata=metadata,
    )
    
    return profile

```
Modality: DOC

├── Patient Module

    └── Modality: Code
```

## TID 1500 Measurement Report Details

### Template Structure

The Comprehensive SR follows DICOM Template ID 1500 "Measurement Report":

- **TID 1500**: Top-level container for measurement reports
- **TID 1001**: Observer context (who performed the exam)
- **Measurement Groups**: One per workflow step with measurements

### Coding Systems Used

| System | Designator | Usage |
|--------|------------|-------|
| DICOM Content Mapping Resource | DCM | Standard DICOM codes |
| SNOMED CT | SCT | Anatomy, procedures |
| CPT-4 | CPT | Procedure codes |
| UCUM | UCUM | Measurement units |

### Standard Codes

| Code | Scheme | Meaning | Usage |
|------|--------|---------|-------|
| 126000 | DCM | Imaging Measurement Report | Root container |
| 125007 | DCM | Measurement Group | Measurement container |
| 112039 | DCM | Tracking Identifier | Group identifier |
| 363698007 | SCT | Finding Site | Anatomical location |
| 121008 | DCM | Person Observer Name | Technologist |
| 121058 | DCM | Procedure reported | Protocol/CPT code |
| 121070 | DCM | Findings | Summary text |
| 121106 | DCM | Comment | Step notes |

### UCUM Unit Codes

Common measurement units mapped to UCUM:
- **cm** → `cm` (centimeters)
- **mm** → `mm` (millimeters)
- **seconds** → `s`
- **Hz** → `Hz`
- **MHz** → `MHz`
- **%** → `%`

### Example 1: Complete Exam Documentation

```python
#!/usr/bin/env python3
"""Complete exam with PDF and SR generation."""

from datetime import datetime
from models import ExamSession, ProtocolWorkflow, Measurement
from protocol_parser import ProtocolParser
from pdf_generator import TechProtocolPDFGenerator

    encapsulate_pdf_to_dicom,

parser = ProtocolParser()

session = ExamSession(
    session_id=f"SESSION_{datetime.now().strftime('%Y%m%d_%H%M%S')}",
    patient_id="PAT_PROD_98765",
    technologist_name="Jones, Sarah",
    exam_start_time=datetime.now(),
    protocol_id=workflow.protocol_id,
    modality=workflow.modality,
    cpt_code=workflow.cpt_code,
)

workflow.steps[2].measurements = [
    Measurement("Liver Span", 15.1, "cm", "12-16 cm", "normal"),
    Measurement("Portal Vein", 10.8, "mm", "<13 mm", "normal"),
]
workflow.steps[2].notes = "Good acoustic window. No abnormalities detected."

# 4. Complete session
session.completed_steps = [s.step_id for s in workflow.steps]
session.summary_notes = "Routine exam. All measurements within normal limits."
session.status = "completed"

# 5. Generate PDF
generator = TechProtocolPDFGenerator()
pdf_bytes = generator.generate_pdf(session, workflow)

# 6. Create DICOM objects and send to PACS

    pdf_bytes, session, workflow,
    send_to_pacs=True
)

sr_ds, _, sr_resp = create_sr_with_measurements(
    session, workflow,
    send_to_pacs=True
)

print(f"✅ Study uploaded: {sr_ds.StudyInstanceUID}")
print(f"   PDF Instance: {pdf_resp['instance_id']}")
```

You already have almost all the plumbing; what’s missing is essentially a **report-template layer that sits on top of the Customized Protocol** and a small adapter into SR.

Given your clarification (Order Form → Customized Protocol → Preliminary Report Template), here’s what’s concretely missing to go from **Customized Protocol** to a **personalized Preliminary Report Template**.

---

## 1. A canonical “Preliminary Report Template” model

Right now you have:

- `ExamSession` + `ProtocolWorkflow` (steps, required_measurements, imaging_views, etc.) that drive the tech side and TID‑1500 SR.
- SR → JSON → ACR enrichment → consistency profile (in `tools/preliminary-report` + `packages/report-consistency`).

What you *don’t* have is a stable, explicit object like:

- `PreliminaryReportTemplate` (or equivalent JSON schema) that defines, for a given protocol:

  - Sections: Indication, Technique, Findings, Impression, Measurements, Recommendations.
  - Per‑field metadata: label, type (free text, numeric, coded list), required/optional, default source.
  - How structured items (measurements, views, key findings) are laid out.

Until that exists, everything after the protocol is “raw SR + analysis,” not an explicit template that a GUI/editor can render and drive.

---

## 2. A library of **report templates keyed by protocol**, not just modality

You have:

- `all_imaging_protocols_301.ndjson` for protocol definitions.
- Tech workflows and SR encoders that know what measurements/views a protocol expects.
- ACR/GraphRAG knowledge tied to indications/protocols.

You’re missing:

- A **report template library** that is indexed by protocol identity (e.g., `protocol_id`, CPT, LOINC/RPID) and possibly modality, for example:

  - `report_templates/US_RUQ.json`
  - `report_templates/US_Renal.json`
  - `report_templates/CT_AbdPelvis_Contrast.json`

Each entry would specify:

- Which sections appear, in what order.
- Which measurements or protocol steps must surface as explicit findings lines (e.g., “Gallbladder wall thickness: __ mm”).
- Which optional findings or checkboxes apply.
- Default phrasing or bullet structures for that protocol.

In WingMan terms, this is essentially the `us_templates.json` + prompts layer generalized and re-homed in `ai-imaging-platform`. That mapping does not exist yet.

---

## 3. A mapping layer: **Customized Protocol → chosen report template**

Your `GraphRAGProtocolSelector` + `ProtocolCustomization` give you a **chosen protocol** with customizations:

- `protocol_id`, `protocol_name`, `modality`, CPT/LOINC (via terminology layer).
- Customizations: modified parameters, safety overrides, equipment adjustments, etc.

What’s missing is:

- A small adapter that, given `(protocol, customization, patient/order context)`:

  1. Resolves the **report-template ID**:
     - e.g., for `protocol_id="US_RUQ_ABD_001"` choose `report_templates/US_RUQ.json`.
  2. Applies **customizations** to the template:
     - Include/exclude sections or fields based on which measurements are actually in the customized protocol.
     - Mark some fields required/optional depending on adaptation (e.g., if you added elastography, add elastography findings slots).
  3. Injects **defaults** from:
     - Order/indication (Indication text, clinical question).
     - SelectionContext (urgency, special precautions).
     - ACR enrichment (topic IDs, guideline references).

That logic currently doesn’t exist; only protocol and workflow selection do.

---

## 4. A “templated SR” view: mapping template fields → TID‑1500 SR structure

You already have:

- `dicom_sr_encapsulator.create_comprehensive_sr_with_measurements(...)` that turns `ExamSession + ProtocolWorkflow` into a TID‑1500 SR.
- SR → JSON → consistency analysis.

But for a **Preliminary Report Template in structured report format**, you also need:

- A clear mapping from **template sections/fields** into SR content items, e.g.:

  - Technique / Protocol → `Procedure reported`, `Findings`, possibly custom text nodes.
  - Each “structured finding slot” in the template → a measurement group or coded TEXT entry.
  - Impression outline slots → TEXT content items that your later narrative generator can fill.

In other words, you need:

- A “template‑to‑SR mapper” that knows, for a given `PreliminaryReportTemplate`, where each field lives in the TID‑1500 tree (or in a parallel JSON structure that is clearly isomorphic to that tree).

Right now, SR is generated **from workflow/measurements**, not from a **report template spec**.

---

## 5. Personalization hooks from Order + Customization into the template

You have the pieces of personalization:

- Patient context (age, sex, risk factors) in the clinical engine / GraphRAG selector.
- Clinical indication and urgency from order/SelectionContext.
- ACR topic enrichment (`acr_topics` array in enriched preliminary report JSON).

What’s missing is:

- The actual code that **writes those into the Preliminary Report Template** as:

  - Pre‑filled Indication text.
  - Technique phrase defaulted from protocol + modality suggestion.
  - “Reason for exam” or “Clinical context” from order + ACR topic metadata.
  - Optional pre‑populated recommendation skeletons aligned with ACR topic.

At the moment, they are used for **selection and enrichment**, not for **seeding a template object** that a GUI/editor or LLM can render.

---

## 6. Glue function: `build_preliminary_report_template(customized_protocol, patient_ctx, order_ctx)`

Putting it together, the one conceptual function you don’t yet have is something like:

- Resolve **report template spec** for this protocol.
- Bind it with:

  - Protocol fields (measurements, views, special maneuvers).
  - Patient/order context (indication, urgency, risk).
  - ACR enrichment (topic_id, panel, key doc links).
  - Any protocol customizations.

- Emit a **structured Preliminary Report Template** object (and optionally an isomorphic TID‑1500 SR skeleton) that everything downstream can work from.

The underlying building blocks (protocol workflows, SR tooling, ACR enrichment, consistency engine, terminology, GraphRAG‑based selection) are in place; **what’s missing is the explicit, protocol‑keyed report-template library and the small adapter layer that turns a Customized Protocol + context into a concrete Preliminary Report Template object.**
